# 当 Claude Code 学会作曲：我在 Godot 里搭了一个多 Agent MIDI 音乐生成系统

> 用 7 个 AI Agent 协作，从一段文字描述到完整的游戏配乐，全程人机协同。
>
> *本文为项目 Preview，Clef 目前仍在积极开发和打磨中，后续将会正式开源发布，敬请期待。*

## 缘起

做独立游戏最让人头疼的事情之一，就是音乐。

你可以从素材网站买几首 BGM，但它们永远不会和你的游戏场景完美契合。你也可以花钱请作曲师，但预算往往不允许反复修改。至于自己学？乐理门槛可不低。

作为一个 Godot 游戏开发者，我一直在想：能不能让 AI 帮我写游戏音乐，而且写出来的是符合我的风格需求，**不求最好，但求能用，可以迭代**的曲子？

这就是 **Clef** 的起点。

## Clef 是什么

Clef 是一套**多 Agent 协作的 AI 作曲系统**，通过 Claude Code 驱动 7 个专业角色协同创作，并配套一个 Godot 4.6 MIDI 播放插件。你只需要输入一段自然语言描述，系统就能自动生成完整的游戏配乐，并输出标准 MIDI 文件。

简单说，你可以这样触发它：

```
/clef-compose 一首地下城探索的BGM，2分钟，循环，神秘感，弦乐为主
```

然后坐下来，喝杯咖啡，等音乐生成出来。

## 它是怎么工作的

### 整体架构

整个系统的架构可以简化为三层：

```
┌─────────────────────────────────────────────────┐
│  你的输入：自然语言场景描述 + 风格偏好            │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  多 Agent 协作层（7个专业角色）                    │
│  Composer → Harmonist → Rhythmist               │
│  Reviewer → Revision → Orchestrator → Leader     │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  技术处理层（Python 工具链）                      │
│  验证 → 合并 → 转换 → 表现力注入                  │
└──────────────────┬──────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────┐
│  输出：标准 MIDI + MP3，可直接在 Godot 中播放      │
└─────────────────────────────────────────────────┘
```

*（三层分别对应：用户交互层、智能协作层、工程输出层）*

### 7 个 Agent，各司其职

这不是一个大模型什么都干，而是 7 个角色分工明确的 Agent 团队：

| 角色 | 职责 | 类比 |
|------|------|------|
| **Composer**（作曲家） | 创作主旋律 | 乐队主旋律手 |
| **Harmonist**（和声师） | 编配和弦与和声填充 | 键盘手 |
| **Rhythmist**（节奏师） | 编写低音线和鼓组 | 贝斯 + 鼓手 |
| **Reviewer**（评审员） | 从 6 个维度打分评审 | 音乐制作人 |
| **Revision**（修正员） | 修正格式错误，不碰创作 | 录音助理 |
| **Orchestrator**（编曲师） | 注入表情、力度、混音信息 | 配器师 / 表情师 |
| **Leader**（指挥） | 调度迭代、管理依赖、决定何时收工 | 音乐总监 |

每个 Agent 都有自己的专业领域知识。比如 Composer 不需要知道鼓谱怎么写，Rhythmist 不需要操心旋律走向——就像一个真正的乐队，各司其职。

### 工作流：从想法到成品

整个作曲过程分为几个关键阶段，中间有两个必须由你亲自确认的检查点：

**Step 0 — 需求解析**

系统会从你的自然语言描述中提取关键信息：场景类型（战斗/探索/菜单/商店等，共 10 种游戏场景模板）、情绪风格、时长、是否循环、目标乐器配置等。你还可以提供一首参考 MIDI 让 AI 「模仿风格」，或指定一个 SF2 音色库让系统根据音色特性来写。

**Step 1a — 生成编曲方案**

Leader Agent 会生成一份计划书，包含调性、速度、拍号、曲式结构（如 ABA）、各段落能量等级、每个声部的乐器配置和音域范围。

> 检查点 1：你会看到完整的编曲方案，确认或修改后才会继续。

**Step 1b — 方向小样**

各 Agent 根据方案创作一段短小样（2-16 小节），经过评审和修正后合并为 MIDI。这是你第一次听到实际的音乐。

> 检查点 2：听完小样后，你可以提出修改意见，系统会迭代调整（**用户反馈最多 10 轮**）。

**Step 2 — 完整创作 + 自动迭代**

所有 Agent 生成完整长度的声部。Leader 根据 Reviewer 的 6 维度评审报告（调性一致性、音域合理性、大跳合理性、时值准确性、声部对齐、声部重叠）来调度修改。旋律和声部之间有依赖关系——和声改了，旋律必须跟着调。**自动迭代最多 3 轮**。

**Step 3 — 表现力注入**

Orchestrator 为音乐添加「呼吸感」。通过 MIDI 控制信号（CC）注入音量变化、声像定位、混响深度、颤音等。弯音信息也会在这一步加入，让弦乐滑音、铜管起音等细节更自然。

最终输出标准 MIDI 文件和 MP3，可直接导入 Godot 使用。

## 不是黑盒：全程人机协同

很多 AI 音乐工具的问题是——你输入一段描述，出来一首歌，好坏全凭运气。中间过程你完全看不到，也改不了。

Clef 的设计理念是 **Human-in-the-Loop**：

1. **编曲方案可审查** — 在写第一个音符之前，先确认调性、速度、配器方案
2. **小样可试听可修改** — 先听方向对不对，不对就调整，不浪费时间
3. **每次迭代可追溯** — 系统会自动备份每个版本（score_v1.abc 到 score_v11.abc），随时可以回退
4. **技术验证有保障** — 每次迭代后自动运行 7 项技术检查（调性、音域、大跳、时值、对齐、重叠、音色甜区），FAIL 级别的问题必须修正才能继续

## 真正的 MIDI，不是音频采样

和很多 AI 音乐生成工具不同，Clef 输出的是标准 MIDI 文件，不是音频。

这意味着什么？

- **完全可编辑** — 你可以在任何 DAW（如 REAPER、FL Studio）或 **Godot 内置的 Piano Roll** 里打开、修改每一个音符（编辑模式计划中）
- **音色自由更换** — 换一个 SF2 音色库，同样的音乐听起来完全不同
- **体积极小** — 一首 2 分钟的 MIDI 文件只有几十 KB
- **实时响应** — 在 Godot 里通过 AudioStreamPlayer 实时合成，无需预渲染
- **CC/弯音/颤音** — 完整支持 MIDI 控制信号，不只是音符 on/off

## 播放引擎：不只是「能播 MIDI」

*（如果你只关心作曲，这部分可以跳过；如果你也想在游戏里实现 MIDI 播放，下面是技术细节。）*

Clef 的 Godot 插件本身就是一个功能完整的 MIDI 播放解决方案：

**SF2 音色库支持** — 内置完整的 SoundFont 2 解析器，读取音色库的采样、ADSR 包络、键盘/力度范围等信息，配合默认的 GeneralUser-GS 音色库开箱即用。

**复音池管理** — 支持最多 128 复音，每个音符独立分配 AudioStreamPlayer，带有完整的 ADSR 状态机。当复音数超出上限时，智能的 Voice Stealing 算法会优先释放低优先级音符。

**实时表现力** — 支持 CC1（颤音）、CC7（音量）、CC10（声像）、CC11（表情）、CC64（延音踏板）、弯音（14 位精度）等控制信号，所有变化实时应用到活跃音符上。

**Clef Station 编辑器** — Godot 编辑器内的集成工作站，包含 Transport Bar（播放控制）、Piano Roll（音符可视化）、Mini Mixer（通道混音）、MIDI Monitor（事件监控）、Soundfont Browser（音色浏览器）。

## 一些设计细节

### 音色库感知

Midi播放需要音色库，而AI作曲家需要把音色库的表达力纳入考虑。Clef有自己的音色库分析脚本，可以生成音色库的专属特性画像。这份画像可以用来分析音色库的物理特性：每个乐器的有效音域、**最佳音区（“甜区”——也就是乐器听起来最好听的音区）**、力度层数、起音/释放时间。然后所有 Agent 在创作时都会考虑这些约束。

比如，音色库里某个弦乐只有 1 层力度采样，Agent 就不会写太多力度变化。某个乐器的起音很慢，Agent 就不会给它安排快速的琶音。这样生成的 MIDI 在用目标音色库播放时效果更好。

### 6 维度评审系统

Reviewer Agent 不是简单地给一个「好/不好」的判断，而是从 6 个专业维度打分：

1. **调性一致性** — 是否始终在目标调性内
2. **音域合理性** — 是否在乐器可演奏范围内
3. **大跳合理性** — 是否有过多不自然的大跳
4. **时值准确性** — 小节时值是否正确
5. **声部对齐** — 多声部的节拍是否对齐
6. **声部重叠** — 声部之间是否有不该有的重叠

每项都有 PASS / WARN / FAIL 三个级别。FAIL 必须修复才能进入下一轮，WARN 会被记录但不阻塞流程。

### 依赖调度

Leader Agent 管理的不是一个简单的线性流程，而是一个有依赖关系的任务图。

比如，如果 Reviewer 指出和声声部有问题，Leader 会先派任务给 Harmonist 修复和声。因为旋律（V:1）依赖和声（V:2），所以 Harmonist 的任务完成后，才能派任务给 Composer 调整旋律。这种「先修地基，再修房子」的调度逻辑，避免了反复修改同一个问题。

## 实际效果

目前系统已经生成了多首不同风格的游戏配乐 demo：

- **DungeonExploration** — 地下城探索风格，神秘弦乐 + 低音铺底
- **SpaceBattle_8bit** — 8-bit 风格太空战斗，芯片音乐感
- **FieldExploration** — 田野探索风格，明亮轻快

每首从初始方向到最终成品都经过 3-5 轮迭代，完整记录了中间版本，可以清晰地看到音乐从「还行」到「能用」的进化过程。

## 技术栈

- **Godot 4.6+** — 游戏引擎，MIDI 播放和编辑器集成
- **Claude Code** — LLM 编排框架，驱动多 Agent 协作
- **Python 3.10+** — 工具链（ABC→MIDI 转换、验证、CC 注入等）
- **music21** — 音乐理论分析库
- **mido** — MIDI 文件处理库
- **ABC 记谱法** — Agent 之间的音乐交换格式

## 写在最后

Clef 目前还在打磨阶段——自动迭代的稳定性、更多游戏场景模板的支持、更丰富的音色库适配，这些都在持续完善中。

但**它不是一个「一键生成完美音乐」的魔法工具，而更像是一个不知疲倦、知识全面但需要指导的虚拟乐队**。

你不需要会乐理——告诉它你想要什么感觉，它会帮你把想法变成五线谱。但如果你有乐理基础，你可以深度参与每一个决策：从调性选择到配器方案，从旋律走向到力度变化。

对于独立游戏开发者来说，这意味着你终于可以在不依赖外部资源的情况下，为你的游戏创作量身定做的音乐。
